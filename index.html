// 01 — overview

What Is This Script?

This is a custom ArduPilot Lua script that adds intelligent, two-phase precision landing behaviour to a QuadPlane VTOL. Instead of relying on ArduPilot's built-in precision landing (which can drift and land slightly off-target), this script takes direct control of the descent rate and waypoint targeting to guarantee that the aircraft only descends when it is properly centred over the landing pad.

🎯

Precision Targeting

Uses a downward-facing vision sensor (IR lock / camera) to continuously calculate the distance from the landing pad and correct the flight path.

🔒

Altitude Lock

Below 7 m, the aircraft freezes its altitude entirely if the target is lost or if XY offset exceeds 35 cm — eliminating off-target touchdowns.

📡

Live GCS Feedback

Sends human-readable status messages to Mission Planner / QGroundControl at every decision point so operators know exactly what the script is doing.

⚙️

Configurable Params

Key values (XY gain, descent radius, hold-on-lost) are exposed as ArduPilot parameters — tunable without editing the Lua source code.

🛡️

Fault-Tolerant Loop

A pcall wrapper catches any Lua runtime errors and logs them, ensuring the script never crashes silently or hangs the vehicle.

🔀

Dual-Mode: QLAND + LOITER

Works automatically during QLAND and optionally activates during QLOITER when a dedicated RC aux switch is flipped to the high position.

// 02 — use cases

Where Is It Used?

This script is designed for any QuadPlane (VTOL) aircraft running ArduPilot Plane firmware that needs to land on a specific, known point with high accuracy. Common real-world applications include:

🚁

Delivery Drones

Land on a marked pad at a customer location rather than drifting onto grass, obstacles, or an incorrect spot.

🚢

Ship / Vehicle Deck Landing

Target a moving or stationary deck marker for safe recovery operations where surrounding space is limited.

🔋

Autonomous Charging Pads

Must connect to charging pins accurately — a few centimetres of error means a failed charge cycle.

🔬

Research / Survey

Repeatability matters — land on the exact same GCP marker between flights to maintain consistent data collection baselines.

⚠ Hardware Requirement: The aircraft must have a downward-facing precision landing sensor (IR-Lock, OpenMV, or similar) connected to the flight controller and configured via PLND_TYPE. Without this, the script has no target to track.

Prerequisites

🖥 Firmware

  • ArduPilot Plane 4.3 or later (Lua scripting API must be available)
  • Vehicle type: QuadPlane (frame class 10 / 13 / etc.)
  • SCR_ENABLE = 1 — enable the scripting engine
  • SCR_HEAP_SIZE — ensure at least 100 KB of heap for scripts
  • PLND_ENABLED = 1 — precision landing subsystem active
  • PLND_TYPE — set to match your sensor (1 = companion, 3 = IR Lock, etc.)

🔧 Hardware Required

  • Flight Controller: Pixhawk 4 / 6C / Cube Orange or any ArduPilot-compatible FC
  • Precision Landing Sensor: IR-Lock + IR Beacon, or monocular camera with ArUco/AprilTag
  • MicroSD Card: Required — scripts are loaded from /APM/scripts/
  • RC Transmitter: One spare channel for the PRECLOITER aux switch (CH6–CH12)
  • Landing Pad: IR-reflective beacon or printed AprilTag marker (size depends on AGL height)

📦 Software / Tools

  • Mission Planner 1.3.80+ or QGroundControl (for parameter setup and GCS messages)
  • ArduPilot Lua scripting API documentation (for future modifications)
  • A text editor (VS Code recommended) to read/modify the .lua file

// 05 — installation

How to Install

  1. Enable Lua Scripting on the Flight Controller In Mission Planner, go to Config → Full Parameter List. Set SCR_ENABLE = 1 and reboot the FC. Also set SCR_HEAP_SIZE = 131072 (128 KB) to give scripts enough memory.
  2. Configure Precision Landing Parameters Set PLND_ENABLED = 1 and PLND_TYPE to match your sensor. For IR-Lock use PLND_TYPE = 3. Reboot after changes.
  3. Copy the Script to the SD Card Remove the SD card from your Pixhawk (or connect via USB mass storage). Navigate to /APM/scripts/ — create the folder if it doesn't exist. Copy vtol_strict_landing.lua into that folder.
  4. Re-insert the SD Card and Power On On boot, ArduPilot will automatically discover and load all .lua files in /APM/scripts/. Watch the GCS messages — you should see: PLND: UPDATED 7m LOGIC LOADED.
  5. Verify Script Parameters Appear After the first boot with the script, new parameters prefixed PLND_ will appear in the parameter list. Confirm PLND_XY_GAIN, PLND_DESCENT_RAD, and PLND_LOST_HOLD are present.
  6. Set the Aux RC Channel Assign your PRECLOITER switch channel to option 39 (e.g. RC7_OPTION = 39). Test that toggling the switch sends HIGH/LOW correctly by watching GCS messages in QLOITER mode.
  7. Bench-Test Target Detection Hold the aircraft over the landing beacon on a bench. Check that GCS shows PLND: TARGET ACQUIRED. Move away and verify PLND: TARGET LOST.
  8. Fly a Low-Risk Test In a safe open area, command a QLAND from 15 m above the beacon. Observe the two-phase behaviour: free descent above 7 m, then strict altitude-hold and centering below 7 m. Verify touchdown is within ≈35 cm.
📁 File Path: The script must be placed at exactly /APM/scripts/vtol_strict_landing.lua on the SD card. Any subdirectory will be ignored by the ArduPilot scripting engine.
// 06 — logic deep-dive

How It Works

The script runs a protected update loop at 10 Hz (every 100 ms). On each tick it reads altitude, checks whether a precision target is visible, and applies one of two rule-sets depending on which phase the aircraft is in.

↑ above 7m

☁ Phase 1 — Approach (Above 7 m)

  • If target is visible → steer toward it using PLND_XY_GAIN amplification on the next waypoint location
  • Normal descent rate is allowed (Q_LAND_FINAL_SPD)
  • If target is lost → just descend normally, no override
  • GCS reports ALT: X.XXm | TRACKING TARGET (>7m)
↓ below 7m

🔒 Phase 2 — Strict Precision (Below 7 m)

  • XY correction is applied every tick regardless
  • Descent ONLY if xy_dist ≤ PLND_DESCENT_RADIUS (default 35 cm)
  • If target lost → set_land_descent_rate(0) — altitude frozen
  • If XY offset > 35 cm → set_land_descent_rate(0) — altitude frozen
  • GCS reports centred/hold status every tick

Decision Flow — Per Tick

update() called @ 10 Hz │ ├── PLND_ENABLED < 1? → ABORT (disabled) │ ├── precision_landing_active()? │ ├── NO → return (wrong mode / switch off) │ └── YES ↓ │ ├── get altitude (AHRS NED) │ ├── update_target() [check precland:target_acquired()] │ ├── acquired → last_seen_time = now │ └── have_target = (now - last_seen_time) < 500ms │ ├── current_alt > 7.0m ? │ ├── YES — PHASE 1 │ │ ├── have_target? YES → steer toward target (gain applied) │ │ │ NO → no override │ │ └── set_land_descent_rate(Q_LAND_FINAL_SPD) → RETURN │ │ │ └── NO — PHASE 2 (strict) │ ├── have_target? NO → set_descent_rate(0) HOLD ALT → RETURN │ │ │ ├── get target XY location │ ├── compute xy_dist to target │ ├── apply XY correction to next_WP (always) │ │ │ └── xy_dist ≤ 0.35m? │ ├── YES → set_descent_rate(Q_LAND_FINAL_SPD) DESCEND │ └── NO → set_descent_rate(0) HOLD ALT

Key Code — Phase 2 Descent Gate

-- Below 7m: only descend when centred
if xy_dist <= PLND_DESCENT_RADIUS:get() then

    vehicle:set_land_descent_rate(Q_LAND_FINAL_SPD:get())

    gcs:send_text(MAV_SEVERITY.INFO,
        "PLND: CENTERED - DESCENT ENABLED (<7m)")
else
    vehicle:set_land_descent_rate(0)   -- altitude frozen

    gcs:send_text(MAV_SEVERITY.INFO,
        string.format("PLND: HOLD ALT (XY=%.2fm <7m)", xy_dist))
end

Configurable Parameters

After the first boot with the script loaded, these parameters appear in Mission Planner's Full Parameter List and can be adjusted without touching the code:

Parameter Default Description
PLND_XY_GAIN2.5Amplifies the XY steering correction. Higher = more aggressive steering toward target. Reduce if oscillating.
PLND_DESCENT_RAD0.35 mMaximum XY distance from target allowed before descent is permitted in Phase 2. Tighten for higher precision.
PLND_LOST_HOLD1Set to 1: freeze altitude when target is lost below 7 m. Set to 0: continue descending even if lost.
PLND_ENABLED(from PLND)Master on/off. If 0, the script exits immediately on load and does nothing.
💡 Tuning Tip: If the aircraft oscillates around the target instead of converging, lower PLND_XY_GAIN from 2.5 to 1.5. If it's slow to converge, increase to 3.0. Never exceed 4.0 — it may cause overcorrection loops.

// 10 — key learnings

What I Learned

Embedded Lua runs differently from desktop Lua

The ArduPilot Lua sandbox exposes a limited, aviation-specific API. Learning which functions exist, when they return nil vs raising errors, and how to safely check return values was the biggest initial hurdle.

Safety logic must be explicitly designed — it doesn't happen automatically

A naive script that always descends will always land somewhere, but not reliably on target. The "freeze altitude" decision for every edge case (target lost, sensor lag, XY drift) had to be explicitly coded rather than assumed.

500 ms target-lost hysteresis prevents false triggers

Without the last_seen_time window, any momentary sensor glitch (a frame drop, brief occlusion) would immediately freeze altitude and interrupt the landing. The 500 ms window smooths this while remaining fast enough to react to real target loss.

pcall is mandatory in a 10 Hz aviation loop

If any Lua error occurs unprotected, the script silently dies and leaves the vehicle without override. Wrapping the entire update in pcall and logging errors means the loop always reschedules — even if one tick throws an exception.

Gain tuning is as important as the algorithm itself

PLND_XY_GAIN = 2.5 works well in light wind, but in gusty conditions it can cause overcorrection oscillations. Exposing it as a parameter rather than a hard-coded constant made field tuning possible without firmware reflashing.

Common Issues & Fixes

❌ Script not loading ("PLND: UPDATED 7m LOGIC LOADED" never appears)

Check: SCR_ENABLE = 1 and the file is at /APM/scripts/vtol_strict_landing.lua (not in a subdirectory). Also verify SD card is formatted FAT32 and the FC can read it (check for SD card errors in boot log).

❌ PLND parameters don't appear after boot

This usually means the script failed to execute (a Lua syntax error). Check the GCS log for PLND ERROR: messages. Also try increasing SCR_HEAP_SIZE — if the heap is too small the script fails silently.

⚠ TARGET ACQUIRED never shows (sensor present, PLND_TYPE set)

Verify the sensor is wired to the correct UART/I2C port and PLND_TYPE matches the sensor driver number exactly. Test the sensor in Mission Planner's Precision Landing live view. Some sensors also need PLND_ORIENT to match the camera mounting direction.

⚠ Aircraft oscillates at target instead of landing

Reduce PLND_XY_GAIN to 1.5–2.0. The default 2.5 can overcorrect in windy conditions or if the sensor has latency. Also consider increasing PLND_DESCENT_RAD slightly to 0.5 m if targeting precision requirements allow it.
